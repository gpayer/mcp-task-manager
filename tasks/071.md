Bug: `get_next_task` currently only returns the next task in state `todo`, which can strand already-started work when another task remains `todo`.

Expected behavior:
- `in_progress` tasks should be prioritized ahead of `todo` tasks.
- If any actionable task is already `in_progress`, `get_next_task` should return that work before selecting a fresh `todo` task.
- This should apply consistently with existing blocking and subtask-selection rules.

Example:
- Parent task `#67` is `in_progress`
- Subtask `#68` is `in_progress`
- Subtask `#69` is `todo`
- `get_next_task` should return `#68`, not `#69`.

This likely affects workflow orchestration because the controller relies on `get_next_task` to resume interrupted work.
