Steps:
1. Refactor `Index.NextTodo()` so actionable tasks are ranked in parent-aware groups instead of treating every subtask as an independent candidate.
2. For each todo candidate, derive a primary sort key:
   - Top-level tasks use their own priority, `CreatedAt`, and ID.
   - Subtasks inherit their parent task's priority, `CreatedAt`, and ID for the first-stage comparison.
   - Subtasks of `in_progress` parents must still be considered before all normal todo candidates, preserving the current workflow rule.
3. After selecting the winning parent/task group, choose the actual returned task inside that group by child-level ordering:
   - For a standalone top-level task, the group winner is the task itself.
   - For a parent group with multiple actionable subtasks, sort those subtasks by their own priority, then `CreatedAt`, then ID, and return the first actionable child.
4. Keep the existing exclusions intact: skip blocked tasks, skip parent tasks that have subtasks, and ignore non-todo entries.
5. Run the targeted regression tests from Task #68, then run the broader storage package tests to confirm the new grouping logic does not break existing `NextTodo` behavior.

Code guidance:
- Implement the ordering in `internal/storage/index.go`; `Service.GetNextTask()` is only a pass-through and should not need behavior changes.
- Prefer a small helper or internal struct for the group sort key so parent-derived and child-derived ordering rules are explicit and testable.
- Preserve deterministic tie-breaking with ID comparisons at both the parent-group stage and the child-within-group stage.
- Avoid broad rewrites: the goal is to change how `NextTodo()` ranks subtasks, not to redesign the index or task model.
