`get_next_task` currently returns the next task in state `todo`, even when there is already actionable work in state `in_progress`.

Observed behavior:
- Parent task `#67` is `in_progress`.
- Subtask `#68` is already `in_progress`.
- Subtask `#69` is still `todo`.
- `get_next_task` returns `#69` instead of surfacing the already-started work on `#68`.

Expected behavior:
- `get_next_task` should prioritize actionable `in_progress` tasks ahead of `todo` tasks.
- In other words, if there is work already in progress, the workflow should continue that task before selecting a new todo item.
- This should apply to subtasks as well, since the orchestrator may resume in the middle of an execution loop after interruption or restart.

Why this matters:
- The current behavior can strand an `in_progress` task and incorrectly advance to a different `todo` task.
- That breaks resumability for interrupted workflow execution and can create inconsistent task state.

Fix scope:
- Update `get_next_task` semantics so resumable active work wins over not-yet-started work.
- Preserve existing priority/ordering rules within each state as appropriate.
