`get_next_task` currently sorts actionable subtasks using the subtask's own priority/creation order instead of first ranking them by their parent task.

Expected behavior:
- A subtask should inherit its parent's priority for the first stage of sorting.
- A subtask should also inherit its parent's age/ID ordering for that first stage.
- After the parent "wins" against other top-level tasks, the parent's children should then be sorted among themselves to choose the true next actionable task.

Example:
- Tasks `#1`, `#2`, and `#3` exist.
- `#1` and `#2` are top-level tasks with equal priority.
- `#3` is a child of `#1`.
- Current behavior: `get_next_task` returns `#2`.
- Expected behavior: `get_next_task` returns `#3`, because `#3` should sort as if its parent `#1` were competing with `#2` first.
- `#3` should still win even if its own priority is lower than `#2`, because the parent task's priority and ordering should be used before child-level ordering.

In short: subtasks should inherit their parent's priority and creation-order/ID position for primary sorting in `get_next_task`; child-specific sorting happens only after the parent has been selected as the winning task group.
