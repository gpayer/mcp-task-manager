When resuming workflow, an already-started task can be stranded because `get_next_task` only returns `todo` tasks. This breaks the execution loop when a subtask is already `in_progress` and should be resumed before selecting new work.

Expected behavior:
- `get_next_task` should prefer actionable `in_progress` tasks over actionable `todo` tasks.
- The same filtering rules should still apply: skip blocked tasks and skip parent tasks with incomplete subtasks.
- Subtasks of `in_progress` parents should still be handled consistently with overall task selection.

Reproduction:
- Parent task `#67` is `in_progress`.
- Subtask `#68` is `in_progress`.
- Subtask `#69` is `todo`.
- Current behavior: `get_next_task` returns `#69`.
- Expected behavior: `#68` should be returned/resumed first.

This should be fixed as a separate bug from the parent-ordering work in Task `#67`.
